1. Project Overview

Student Bug Tracker is a full-stack web application built for student teams, hackathons, and open-source projects. It combines project management, bug tracking, real-time collaboration, and analytics into one platform.

Bug Tracking

📋

📊

Analytics

Charts, completion rates, productivity trends

🔔

Notifications

In-app and email alerts for assignments and deadlines

🛡️

Admin, Developer, and Tester roles with different permissions

4. Features in Detail

🔐 Authentication

The app uses JWT (JSON Web Token) authentication. When you log in, the server creates a signed token that expires in 7 days. This token is stored in your browser's localStorage and sent with every API request.

Enter email + password
→
Backend verifies password (bcrypt)
→
JWT token created
→
Token stored in localStorage
→
All future requests use token
ℹ️ Why JWT? JWT is stateless — the server doesn't need to store sessions in a database. The token itself contains the user ID and role, which the server verifies using a secret key.

8. User Flows

Flow 1: Admin Creates a Project and Invites Team

1. Admin registers
→ Selects "Admin" role during registration
2. Goes to Projects
→ Clicks "New Project", enters name and description
3. Copies invite code
→ UUID like: 539c5256-5439-4834-b53a-c90f00d53920
4. Shares with team
→ Sends invite code via WhatsApp, email, etc.

Flow 2: Developer Joins and Works on Tasks

1. Developer registers
→ Selects "Developer" role
2. Goes to Projects
→ Clicks "Join Project", pastes invite code
3. Opens Kanban
→ Sees all tasks, finds tasks assigned to them
4. Drags task
→ Moves to "In Progress", team sees it instantly
5. Fixes bug
→ Opens bug, adds comment, clicks "Mark as Resolved"

Flow 3: Tester Reports a Bug

1. Tester registers
→ Selects "Tester" role
2. Joins project
→ Uses invite code from Admin
3. Goes to Bug Reports
→ Clicks "Report Bug"
4. Fills in details
→ Title, description, steps, screenshot
5. AI analyzes
→ Clicks "AI Analyze Severity" for auto-detection
6. Submits
→ Bug appears in list, assigned developer gets notified

Render free tier sleeps after 15 minutes of inactivity. The backend pings itself every 14 minutes to stay awake:

  https.get(`${RENDER_EXTERNAL_URL}/api/v1/health`, ...);
}, 14 * 60 * 1000);

12. Q&A — Common Questions

Q: What is this project?
A full-stack web application for student teams to track bugs, manage tasks on a Kanban board, collaborate in real-time, and analyze project progress. It has three user roles (Admin, Developer, Tester) with different permissions.
Q: What tech stack does it use?
Frontend: Next.js 16 (React), TypeScript, Tailwind CSS, Zustand, Socket.io-client, Recharts
Backend: Node.js, Express, TypeScript, Prisma ORM, Socket.io, JWT, bcrypt
Database: PostgreSQL 16
Hosting: Render (free tier)
Q: How does authentication work?
JWT (JSON Web Token) authentication. When you log in, the server verifies your password using bcrypt, then creates a signed JWT token that expires in 7 days. This token is stored in localStorage and sent with every API request in the Authorization header. The server verifies the token on every protected endpoint.
Q: How does real-time collaboration work?
Socket.io creates a persistent WebSocket connection between the browser and server. When a user moves a task on the Kanban board, the frontend emits a "task-moved" event to the server. The server broadcasts this to all users in the same project room. Their browsers receive the event and update the UI instantly — no page refresh needed.
Q: What is Prisma?
Prisma is an ORM (Object-Relational Mapper). Instead of writing raw SQL, you write TypeScript code like prisma.bug.findMany({ where: { projectId } }) and Prisma translates it to SQL. It also manages database migrations — when you change the schema, Prisma generates and applies the SQL changes automatically.
Q: What is Zustand?
Zustand is a state management library for React. It stores global application state (current user, projects, tasks, bugs) in memory. When the state changes, all React components that use it automatically re-render. It also persists some state to localStorage so it survives page refreshes.
Q: Why is the first login slow?
The backend runs on Render's free tier, which spins down after 15 minutes of inactivity. The first request wakes it up, which takes 30-60 seconds. After that, it runs at full speed. A keep-alive ping runs every 14 minutes to prevent this during active use.
Q: How does the invite code system work?
When an Admin creates a project, PostgreSQL automatically generates a UUID (e.g., 539c5256-5439-4834-b53a-c90f00d53920) as the invite code. The Admin copies this and shares it with team members. When a Developer or Tester enters this code in the "Join Project" dialog, the backend finds the project by invite code and adds them as a member.
Q: What does the AI severity analysis do?
It's a keyword-based analysis (not a real AI model). The backend scans the bug title and description for keywords. Words like "crash", "data loss", "security" → Critical. Words like "error", "fail", "broken" → Major. Everything else → Minor. It also suggests tags like "authentication", "api", "mobile" based on keywords.
Q: Where is data stored?
All data (users, projects, tasks, bugs, comments, notifications) is stored in PostgreSQL on Render. The in-memory cache is temporary and clears on server restart. Time tracking entries are stored in browser localStorage (not the database).
Q: What are the database indexes for?
Without indexes, every database query scans the entire table row by row — slow when there are thousands of records. Indexes are like a book's index — they let the database jump directly to the relevant rows. We added 18 indexes on the most common query patterns (e.g., "get all bugs for project X with status OPEN").
Q: What is the in-memory cache?
A simple TTL (time-to-live) cache inside the backend process. When multiple users request the same data within 30 seconds, the second request is served from memory instead of hitting the database. This reduces database load under multi-user scenarios. It's automatically invalidated when data changes (create/update/delete).
Q: Does it work without Redis?
Yes. Redis is optional. The app uses an in-memory cache as a fallback. Redis would provide a shared cache across multiple server instances, but since we run a single instance on Render free tier, in-memory cache is sufficient.
Q: Does it work without email (SMTP)?
Yes. Email is optional. In-app notifications (the bell icon) work without SMTP. Email notifications (sent to your inbox) require SMTP configuration with a Gmail App Password.
Q: How is the frontend deployed as a static site?
Next.js has an output: 'export' option that generates plain HTML, CSS, and JavaScript files (no server needed). These files are uploaded to Render's static site hosting. The browser downloads them and runs the React app entirely client-side. API calls go to the separate backend service.
Q: What is render.yaml?
A blueprint file that defines all Render services in code. It specifies the backend web service, frontend static site, and PostgreSQL database — including build commands, environment variables, and configuration. When connected to GitHub, Render reads this file to set up the deployment automatically.
Q: How do I add a new team member?
1. Admin copies the invite code from the Projects page (visible only to admins)
2. New member registers at /register with Developer or Tester role
3. New member goes to Projects page → Join Project → pastes the invite code
4. They are now a member and can see all project data
Q: What is the project structure?
backend/src/index.ts — All API routes and Socket.io handlers (main file)
backend/src/server.ts — Entry point with keep-alive and graceful shutdown
backend/prisma/schema.prisma — Database schema definition
frontend/src/app/dashboard/ — All dashboard pages
frontend/src/lib/store-api.ts — Zustand store with all API calls
frontend/src/lib/api.ts — Axios instance with JWT interceptor
frontend/src/lib/socket.ts — Socket.io client connection
